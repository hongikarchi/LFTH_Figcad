# 포지셔닝 — MCP 시대 Figcad 가치 + 모델링 범위 결정

> 2026-06-18 전략 정리. 트리거 질문: **"Revit·CAD·Rhino·SketchUp 다 MCP가 잘 돌면, 우리 프로그램 가치는 뭐냐? 모델링 필요한가? 사실상 viewer 아닌가?"**
> 결론 = viewer 아님. 모델러도 아님. **편집 가능한 중립 조율 허브(editable coordination hub).** 모델링은 *입력 UI 가볍게 + 파라메트릭 어휘 풍부하게* 동시 유지. 진짜 미빌드 가치갭 = 멀티모델 라이브 허브.
> 페어 문서: `hub-benchmark-review.md`(경쟁 벤치마크) · `federation-design.md`(허브 스펙) · `ROADMAP.md` 정체성 섹션.

---

## 1. 위협 — MCP가 모든 데스크톱 툴에 깔리면?

곧 Revit·Rhino·SketchUp·CAD 전부 AI 제어 채널(MCP) 생긴다. 그러면 **"AI 편집 잘함"은 해자가 아니다** — MCP 자체가 AI를 commodity로 만든다. 우리가 AI축을 차별점으로 내세우면, 경쟁이 같은 채널을 공짜로 얻는 순간 무너진다.

**따라서 3축(웹·실시간·AI)은 동등하지 않다.**
- **AI** = 곧 table-stakes. 잘하면 되지만 *방어선은 아님*.
- **실시간 + 웹** = 데스크톱 툴에 끼워넣기 구조적으로 어려움 → 방어 가능.

## 2. 진짜 해자 = 실시간 + 웹 + **중립**

| 축 | 왜 데스크톱이 못 따라오나 |
|---|---|
| 실시간 멀티플레이어 (Yjs CRDT 라이브 머지) | Revit 워크셰어링 = 센트럴파일 체크아웃(라이브 아님). Rhino = 멀티플레이어 없음 |
| 웹 / iPad Pencil | 라이선스·설치 0. 외부 누구나 즉시 본다 |
| **중립 (Switzerland)** | Autodesk도 웹뷰어 있음(ACC) — 근데 *영원히 Revit 편애*. **독립툴만 모든 툴 위에 중립** |

→ 정직하게: 웹+실시간 *단독*은 무적 해자 아님(ACC가 웹+뷰어+일부 멀티 보유). **결정적 방어선 = 중립성 + "조율 대화가 이미 벌어지는 자리"**. 오늘 그 조율은 Navisworks + BCF + 이메일 + 스크린샷 + Zoom에 흩어져 있다. 그걸 한 라이브 웹 면으로 모은다.

## 3. MCP 반박 (한 줄)

> **MCP = agent→app 제어면. Figcad = human↔human 조율면**, 중립 머지 상태 위에서. **다른 레이어.**

- MCP를 Revit에 꽂으면 Revit이 *더 좋은 섬*이 된다. 섬은 안 사라진다 — 툴마다 **실행+라이선스 필요**. 4개 툴 조율 = 라이브 데스크톱 앱 4개 동시 구동(비싸고 깨지기 쉬움).
- MCP는 **공유 시각면·presence·CRDT 멀티플레이어 머지를 안 준다.**

## 4. 핵심 — "모델링 필요한가"는 한 덩어리 질문이 아니다. 셋으로 쪼개라

| 층 | 방향 | 근거 |
|---|---|---|
| **수동 입력 UI** (검볼 move/rotate·그립·빠른드로우) | **가볍게 유지·동결** | 정밀 모델링 아님 = 조율급 nudging. iPad에서 손으로 하는 게 말로 하는 것보다 빠른 것만 |
| **파라메트릭 모델 어휘** (kind·op 커버리지) | **풍부하게 유지·확대** | ← **이게 AI 천장.** "말로 하는 건축가"는 변경이 *실제 파라메트릭 op로 표현될 때만* 작동. kind 빈약 → AI 할 수 있는 게 빈약 |
| **정밀/최종 지오** (필렛·디테일·납품도면) | 데스크톱 몫, 커넥터 왕복 | Figcad 일 아님 (LOD 100~250) |

**딜레마 해소:** AI축과 "가벼운 모델링"은 겉으론 반대로 당긴다. 동시에 잡으면 안 부딪힌다 →

> **입력 UI는 가볍게(검볼만) + 파라메트릭 모델 어휘는 풍부하게.**

수동으로 정밀하게 안 한다 ≠ 모델이 빈약해도 된다. 모델 어휘는 AI가 쓰는 substrate니까 *더* 풍부해야 한다. 단지 그 풍부함을 **수동 UI가 아니라 sketch/text→AI 채널로** 노출한다.

**구분 (중요):** per-kind 레지스트리 **리팩터/dedup** = 엔진 건강(유저가치 아님). kind **커버리지 확대** = AI 천장 = 유저가치. 후자는 계속 키운다.

## 5. Viewer인가? — 아니다

- 뷰어는 **되받아쓰기를 안 한다.** Figcad 편집은 커넥터로 원본에 **왕복**(`?op=apply` 라이브쓰기). → 뷰어 아님. **편집 가능한 조율면.**
- 명시적 대척점 = **Speckle**(로드맵 벤치마크). 걘 뷰어+코멘트. 우리는 **편집가능 + AI + 진짜 멀티플레이어**로 이긴다.
- 선례 = **Figma**: 뷰어 아니고, Illustrator급 정밀도도 안 노림. **협업 표면.** 정확히 그 자리.

## 6. 그래서 진짜 만들 가치 = 모델링 깊이 아님, **멀티모델 라이브 허브**

정직한 자기진단 (2026-06-18 코드 조사 기준):
- federation / `ReferenceLayer.ts` = **dev-flag, 프로덕션 UI 없음** (콘솔 `__figcad.referenceLayer`만)
- import는 문서를 **교체**한다 = 한 번에 모델 **하나**

→ 정체성 핵심 약속 — **Rhino + Revit + CAD가 한 화면에 라이브로 *함께*** — 가 거의 안 지어졌다. 반면 모델링-kind 작업엔 손이 많이 갔다. **가치 갭은 모델링 깊이가 아니라 여기다.**

비용 정직하게: 허브는 **커넥터 폭**도 필요(지금 Rhino 전용·수동 ScriptEditor 붙여넣기). 모델링 폴리시보다 큰 짐.

## 7. 권고 (한 방향)

모델링 깊이 추격 멈추고 — **멀티모델 라이브 허브를 실제로 켠다.**

1. **`ReferenceLayer` dev-flag → 진짜 패널 UI** (소스 추가/토글). import-교체 대신 **여러 모델 겹쳐보기**.
2. 그 위에 코멘트·sketch·검볼 = 이미 있는 것들이 비로소 "허브" 위에서 의미가 생긴다.
3. 커넥터 왕복 폭은 그다음 (Rhino 자동시작 → IFC live).
4. 수동 모델링 = 검볼+그립 선에서 **동결**. 나머지 모델 변경 = sketch/text→AI. 파라메트릭 **kind 커버리지는 계속 확대**(AI 천장).

## 8. 인제스트 · 동기화 아키텍처 (2026-06-18 결정)

트리거: "link 걸 때 clean-up(스냅 안맞음·겹친선·문제 지오·data table·레이어정리·labeling)을 native plugin으로? 우리껄로 import 후 별도 페이지로? / link 후 자동연동? git식 push+sync? 실시간 굳이? / 이거 Speckle+AI 아님?"

### 8.1 clean-up = 우리 staging 페이지 (native plugin 아님). 단 2층 분할
모델링처럼 clean-up도 둘로 쪼개야 답이 선다:

| 층 | 위치 | 두께 |
|---|---|---|
| **추출 충실도** (레이어·블록·data table·네이밍 보존) | **커넥터 경계 = at-source** | 얇게, 단 **메타데이터 풍부** |
| **clean-up 지능** (스냅수정·겹침 dedup·분류·labeling) | **중앙 = staging 페이지, AI** | 두껍게 |

**결정 논거:** data table·레이어정리·labeling은 *네이티브 메타데이터*에 의존 — 테셀레이션이 그걸 파괴. ⇒ 커넥터는 **순수 지오 운반이 아니라 시맨틱 구조를 통과**시켜야 하고, clean-up은 **중앙 1곳**. native plugin이 clean-up하면 ① AI를 툴마다 재구현 ② 여전히 라이선스+앱 실행 필요 → 둘 다 패. (clean-up = ingest-time "lifting", brep→파라 시맨틱 리프팅과 같은 레인. roadmap v1.5.)

### 8.2 하나의 primitive — ingest = PR
clean-up·동기화·리뷰는 따로가 아니라 한 흐름:

> **import = PR 올림 · staging clean-up = diff 리뷰 · 승인 = live 모델에 merge.**

"별도 페이지" = 리뷰 게이트 = git push와 같은 primitive. M6 git식 버전관리가 토대.

### 8.3 동기화 — 두 경계, 두 방식 (실시간을 *옮긴다*, 안 버린다)

| 경계 | 방식 | 이유 |
|---|---|---|
| **툴 ↔ 허브** | **git식** (의도적 publish + 버전) | 데스크톱 WIP 라이브 스트림은 *적극적으로 나쁨* — 반쯤 깨진 편집중 지오가 조율판에 흘러감. 준비됐을 때 깨끗한 스냅샷 push가 맞음 |
| **사람 ↔ 허브** | **실시간 CRDT 멀티플레이어** ← 해자 | iPad에서 라이브로 같은 클래시 가리키며 조율하는 *회의* |

**"실시간 굳이?" 답:** 툴 동기화엔 불필요, 사람↔사람 조율엔 필수. 실시간은 *버리는* 게 아니라 *제 위치(사람층)로* 보내는 것. (함정 = 여기서 실시간을 통째로 접으면 §2 해자 반납.)

### 8.4 "Speckle + AI 아님?" — backbone은 맞다, 그게 라인이다
- **backbone(커넥터→버전스냅샷→중립 웹층→diff) = Speckle 맞음. 그건 검증**(F4 = "우리 설계 옳음 검증"). 부끄러움 아님.
- **날카로운 선: Speckle = 파이프(pipe).** Figcad 가치가 파이프면 진다(Speckle 커넥터 수년 앞섬 = 비싼 짐). Figcad는 **파이프 *위에서* 이긴다** = editable + 왕복 + 라이브 멀티플레이어 + AI-native.
- **Speckle은 구조적으로 편집 불가**(뷰어+코멘트+데이터) = **카테고리 갭, 기능 갭 아님.**
- ⇒ **"Speckle+AI"는 editable·멀티플레이어가 *실재할 때만* 과소평가.** `Speckle backbone + AI 챗 사이드바 + 읽기위주`로 끝나면 → **진다. 그게 라인.**

### 8.5 커넥터 폭 긴장 해소 + 안 해본 옵션
- §6 "커넥터 폭 필요" vs 8.4 "out-Speckle 하지 마라" 화해 = **충분한** 커넥터(Rhino·IFC·아마 Revit)면 됨. *망라*는 Speckle 게임, 안 따라감.
- **옵션 (확인 필요·단언 아님):** Speckle 오픈소스 + 공개 stream API → **Speckle stream을 Figcad 소스 하나로 ingest**하면 커넥터-폭 비용 증발 가능. 큰 지출 직격 — API 표면부터 조사할 가치.

### 한 문장 요지
> Figcad = **BIM 조율의 Figma** — 뷰어 아님(되받아씀), 모델러 아님(정밀은 데스크톱). 해자 = 실시간·웹·**중립**. AI는 table-stakes(잘하되 방어선 아님). 모델링은 *입력 가볍게 + 파라메트릭 어휘 풍부 + 멀티모델 허브가 진짜 일감*. 인제스트 = **PR primitive**(import→staging AI clean-up→merge), 동기화 = **툴↔허브 git + 사람↔허브 실시간**. Speckle backbone은 검증 — **단 파이프 *위* editable·멀티플레이어가 실재해야 이긴다.**
