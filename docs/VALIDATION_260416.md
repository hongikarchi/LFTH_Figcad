# Task E 검증 — 260416 MODELING.3dm (436MB) via 커넥터

실제 대형 프로젝트 파일(260416 = 147×170m 경기장, `관중석`·`LOUVER` 등)로 인터롭·속도·안정성 검증.
방법: **M10 커넥터 경로** (Rhino MCP에서 RhinoCommon `File3dm.Read` → subset 추출 → 배포 D-1 `?op=apply` Push).

## 핵심 결과 (성공)

| 항목 | 측정 |
|---|---|
| 브라우저 WASM 통짜 import | ❌ **불가** (탭 메모리 ~200-300MB 캡 < 436MB) — 사전 가설 확인 |
| 커넥터 `File3dm.Read` (436MB) | ✅ **4.0초** (활성 문서 미적재 → MCP 전송 안전, 메모리 OK) |
| 파일 내용 | 24,819 객체 · 113 레이어 · mm. **Brep 17,918(72%)** · 곡선 5,090(20%) · Extrusion 1,075 · 블록 672 |
| 곡선 레이어 | LOUVER 4700 · **GRID 159** · 구로철판 137 · steel stairs 33 · RAIL 18 … |
| GRID subset 추출 | 111 유효 그리드선 (48 퇴화<100mm·72 호/비선 = **우아하게 스킵+카운트**, 크래시 없음) |
| Push → 프로덕션 D-1 | ✅ `200 applied=111 failed=0` in **1.52초** |
| Figcad 반영 | ✅ 111 그리드 = 147.1×170.5m, 좌표 정확, 자동 라벨(A/1/B/2…). `?p=val416-grid` |

→ **브라우저가 못 여는 416MB를 커넥터가 4초에 읽고 매핑 가능분을 1.5초에 Figcad로.** 정체성("상류 인터롭 오케스트레이터") 실증.

## 갭·한계 (정직하게)

1. **72% Brep = 시맨틱 변환 필요(v1.5).** v1 커넥터는 곡선만 매핑. 이 모델의 벽은 Brep이라(벽-곡선 레이어 없음) 자동 추출 안 됨 — brep→파라메트릭 벽 = AI 시맨틱 리프팅(v1.5). 그리드(곡선)는 완벽 매핑.
2. **>2000 ops = 배치 필요.** D-1 바운드(ops≤2000). LOUVER 4700 등 대량 레이어는 분할 POST → 커넥터에 배치(1500/회) 추가(`FigcadConnector.cs`).
3. **레이어 관례 의존.** GRID·Wall Axis·Slab 등 약속 레이어만 매핑. 임의 레이어/형상은 스킵.

## 안정성·속도

- read 4.0s + extract + push 1.5s = 그리드 subset 왕복 **~5.5초**. applied 111/111(실패 0). DoS 작업예산·바운드 정상.
- 활성 문서를 안 건드리는 `File3dm.Read` 패턴이 MCP/메모리 안전의 핵심(활성 416MB는 MCP 전송 파손).

## 결론

브라우저 통짜 import는 비현실(메모리 캡) — **커넥터가 대형 네이티브 파일의 올바른 경로**임을 실증.
파라메트릭 가능분(그리드 등)은 즉시 무손실 이전, Brep 매스는 v1.5 시맨틱 리프팅. 펀치리스트 = 위 갭 3건.
