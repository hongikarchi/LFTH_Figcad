# R1 — coordination-free 파라메트릭 머지 스파이크 결과

> 연구 스파이크(geometry-representation-study.md §6 0단계 / §5 🔴BLOCKING #1). **측정 결과지 증명 아님** — 합성 편집 시나리오 기반. 하니스 = `packages/core/test/merge-spike.spike.test.ts`(throwaway, 빌드 게이트 제외). 2026-06-19, Claude 자율 빌드 중 인라인 수행(배경 에이전트 2회 API 실패 후).

## 질문
Figcad의 핵심 베팅 = **coordination-free 머지 + 사후 lint 플래그**(경로 A, M13-B가 알림 절반 구현). Onshape는 *바로 이걸 못해서* 서버권위+수동해결을 택했다. 측정: N명이 동시 편집 → Yjs CRDT 머지 시 "구조적으로 유효-합의지만 도메인-무효"(겹침·고아개구부·미스핏)가 **얼마나 자주, 그리고 lint가 잡나/놓치나**?

## 방법
카테고리별 100 trial. 매 trial: 시드+기준 벽 2개 → 2 fork(독립) → 각 피어 편집 → **각 피어 솔로 lint(머지 전) 기록** → 상호 머지 → 머지된 문서의 lint findings 중 *어느 피어 솔로에도 없던* 것만 `code+sorted(elementIds)` 키로 "머지 유발 무효"로 집계. 정수 난수(측정 하니스라 결정성 무관).

## 결과 (100 trial/카테고리)

| 카테고리 | 머지유발 무효 | 잡힌 lint 코드 |
|---|---|---|
| **A** 한쪽만 끝점 이동(상대 idle) | **0%** | — |
| **B** 양쪽이 같은 영역에 벽 추가 | **100%** | overlap-wall · unjoined-endpoint · (duplicate) |
| **C** 한쪽 벽 삭제 vs 다른쪽 그 벽에 개구부 | **100%** | orphan-opening (삭제승) |
| **D** 한쪽 벽 단축 vs 다른쪽 먼 위치 개구부 | **100%** | opening-misfit |
| **E** 직교 필드(height vs 끝점) LWW | **0%** | — |

## 판정 — **경로 A 생존, 단 조건부 (lint 플래그 필수)**

1. **"드물다"는 거짓.** 두 사람이 *관련된* 요소를 동시에 편집하면(B·C·D) 무효 수렴은 드문 게 아니라 **거의 확실**(100%). geometry-study §5 #1의 우려가 실측 확인됨 — 순수 coordination-free 머지에 "보통 괜찮겠지"는 안전하지 않다. (단 100%는 *내가 충돌을 의도 설계*한 값 — 실사용 빈도는 "사람들이 얼마나 자주 관련 요소를 동시 편집하나"에 의존. 이 스파이크가 답하는 건 *충돌 클래스별 무효율*이지 전체 머지 무효율이 아님.)
2. **그러나 무효 수렴은 100% 결정적 lint가 잡았다 — 조용한 케이스 0.** overlap-wall·unjoined-endpoint·orphan-opening·opening-misfit가 전부 발화. → **경로 A는 "자동 머지 + 항상 플래그 + 사람 해결"로 살아남는다.** "자동 머지 + *침묵*"은 무효 모델을 조용히 누적 = 진다.
3. **직교 편집은 깨끗(E=0%).** field-LWW가 의미상 무관한 동시 편집(높이 vs 끝점)은 자동으로 옳게 머지. Figcad 불변②의 필드단위 LWW가 *여기선* 충분.

→ **이 스파이크가 M13-B(협업 병합 lint 알림)를 *선택이 아니라 필수*로 격상한다.** 경로 A의 안전성은 머지의 무해함이 아니라 **검출의 신뢰성**에서 온다. M13-B 없이는 경로 A = 조용한 무효 누적.

## 해소 UX 권고 (§5 #2 — "수렴했으나 무효")

무효가 잦으므로(차단 안 함) 플래그·해소가 load-bearing:
- **flag-not-block 유지** (M13-B 현재 방향) — 머지는 LWW로 이미 적용, 차단 불가(coordination-free의 본질). 알림이 1선.
- **단 알림을 눈에 띄게.** 무효가 잦다 = 배너가 자주 뜬다 = 무시 학습 위험. 원격 머지로 새 error(겹침·고아)가 생기면 패널 닫혀 있어도 보이는 글로벌 인디케이터 고려(M13-B 후속, 현재는 패널 내 배너).
- **카테고리별 해소 힌트**: orphan-opening·opening-misfit은 이미 lint fix(삭제)가 있음 → 원클릭. overlap-wall·unjoined-endpoint는 fix 없음(검토) → 끝점 스냅 보조나 소프트락 강화가 v1.5 후보.
- **소프트락 = 예방 레이어**(이미 보유, presence). 같은 요소 동시 편집을 *줄이면* B·C·D 빈도가 준다 — 차단 아닌 마찰. 단 offline-divergent엔 무력(소프트락은 라이브만).
- **3-way/feature merge는 과잉**(F2): CRDT가 라이브를 자동 머지하므로 feature-merge UX는 offline-divergent 버전에만 의미. 이 스파이크는 라이브 동시편집을 측정 — 거기엔 flag+해소로 충분, F2 불필요 확인.

## 한계 (정직)
- 합성 편집(내가 충돌을 설계) — 실사용 동시편집 패턴·빈도 아님. %는 "이 충돌 클래스 주어지면 무효율"이지 전체 머지 무효율 아님.
- 카테고리 A는 퇴화 시나리오(한쪽 idle = 진짜 동시성 아님) → 0%는 "한쪽 편집의 무효는 그 피어 before-set에 이미 있어 머지가 새로 만든 게 아님"을 보일 뿐.
- lint 커버리지에 의존 — lint가 못 잡는 무효 클래스가 있다면 이 스파이크도 못 봄(검출률 100%는 *측정된 4 코드 범위 내*). 새 무효 클래스 출현 시 lint 확장 필요.
- 2-피어만. 3+ 피어 동시 충돌은 미측정(더 나쁠 것).

## 한 줄
**coordination-free 파라 머지는 무효 수렴이 잦지만(관련 동시편집 시 ~100%) 전부 결정적 lint가 잡는다 → 경로 A는 "머지+항상 플래그+사람 해결"로 생존, M13-B가 그 필수 절반. 서버권위 후퇴 불필요. 단 플래그 가시성·해소 UX가 진짜 일감.**
